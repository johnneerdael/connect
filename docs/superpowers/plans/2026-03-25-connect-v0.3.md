# Connect v0.3 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add `connect doctor`, resumable single-file copy with better reporting, and saved foreground local forwarding including local TCP forwards and local SOCKS5 proxies.

**Architecture:** Extend the existing profile-centric CLI and shared SSH session bootstrap instead of introducing a second connection model. Store forward definitions in SQLite beside profile metadata, keep doctor as a non-destructive diagnostics layer over the existing app/SSH context, and layer copy resume/progress behavior onto the current SFTP transfer path.

**Tech Stack:** Rust stable, `clap`, `tokio`, `russh`, `russh-sftp`, `rusqlite`, `keyring`, `directories`, `thiserror`, `assert_cmd`, `predicates`, `tempfile`, std/tokio networking primitives

## Execution Notes

- Work on `codex/connect-cli`; do not create a worktree because the user explicitly asked to work in the main repo.
- Follow TDD for every behavior change: add the failing test first, verify it fails for the expected reason, then write the minimum code to pass.
- Keep the existing shared SSH/auth/host-key flow intact. Doctor, copy, and forward must all reuse the same connection rules.
- Avoid background daemon logic. Forwarding is foreground only in this plan.
- Do not implement remote forwards, recursive resume, or `~/.ssh/config` import.
- Keep the top-level positional `Cli.profile` alias limited to `connect <profile>` only. `doctor` and `forward` must parse exclusively through explicit subcommands so clap does not create ambiguous routing.
