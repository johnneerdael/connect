# `connect` v0.3 Design: Doctor, Resumable Copy, and Local Forwarding

## Summary

`connect` v0.3 adds three operator-facing capabilities that build directly on the existing profile-based SSH client:

- `connect doctor` for local and live diagnostics
- resumable single-file copy with clearer progress and summaries
- saved foreground local forwards, including local TCP forwards and local SOCKS5 proxies

The design keeps the current shared SSH bootstrap intact so host-key trust, auth mode selection, keepalives, and session lifecycle remain consistent across `open`, `exec`, `copy`, and `forward`.

## Goals

- make field debugging faster without asking users to infer failures from low-level SSH errors
- improve large file transfer ergonomics without introducing risky implicit resume behavior
- let users define repeatable tunnels in the same profile-centric model already used for connections
- preserve current CLI ergonomics and avoid introducing daemon/process-manager complexity

## Non-Goals

- no background tunnel daemons or PID/state management
- no remote forwarding or reverse tunnels
- no automatic recursive resume for directory trees
- no destructive repair actions in `doctor`
- no `~/.ssh/config` import/export work in this milestone

## CLI Design

### Doctor

Add:

- `connect doctor`
- `connect doctor <profile>`

Behavior:

- `connect doctor` runs local environment and configuration diagnostics only
- `connect doctor <profile>` runs the same local checks plus live checks against the named profile

Output:

- human-readable by default
- line-oriented check results with status markers and short explanations
- non-zero exit on failed checks

### Copy

Extend:

- `connect copy [--recursive] [--resume] [--progress] <source> <destination>`

Behavior:

- `--resume` is explicit and applies only to single-file upload/download
- `--progress` forces progress output even where auto-detection would stay quiet
- default behavior remains a normal overwrite-style copy

### Forward

Add a new subcommand family:

- `connect forward add <profile> <name> --local <bind_host:bind_port:target_host:target_port>`
- `connect forward add <profile> <name> --socks <bind_host:bind_port>`
- `connect forward list <profile>`
- `connect forward remove <profile> <name>`
- `connect forward run <profile> <name>`
- `connect forward run <profile> --all`

Behavior:

- saved forwards are attached to a profile
- `run` starts in the foreground and keeps the process attached until interrupted or the SSH session dies
- `--all` starts every saved forward for a profile in one attached process

## Detailed Behavior

### Doctor checks

#### Local-only checks

- app directory resolution
- database open/read/write sanity
- secret backend initialization
- profile lookup when a profile is supplied
- profile metadata validity
- auth-mode consistency with stored secrets
- imported private key presence
- SSH agent availability for the current platform/session
- saved host-key presence for the profile, if applicable

#### Live profile checks

When a profile is provided, `doctor` additionally performs:

- DNS resolution or host parsing sanity
- TCP reachability to host and port
- SSH handshake success
- host-key verification result
- auth viability check based on configured auth mode

`doctor` must not open an interactive shell or mutate remote state.

#### Exit semantics

- exit `0` when all checks pass
- exit non-zero when any required check fails
- local-only mode and profile mode use the same result model

### Copy resume and reporting

Resume scope for v0.3:

- supported for single-file upload
- supported for single-file download
- rejected for recursive directory copies

Resume behavior:

- determine existing destination length
- start transfer from the current destination offset
- reject resume when destination is larger than source
- report resumed byte count and final total in the summary

Progress/reporting behavior:

- remain quiet by default in non-interactive contexts
- show progress automatically on TTY or when `--progress` is set
- print a final transfer summary for success
- produce clearer errors for unsupported resume cases and partial directory failures

Future-friendly note:

- content hashing and safe recursive resume are intentionally postponed

### Forwarding behavior

#### Local TCP forward

- listen on the configured local bind address
- for each accepted local socket, open a remote connection to the configured target host/port through SSH
- proxy bytes bidirectionally until EOF or error

#### Local SOCKS5 forward

- listen on the configured local bind address
- accept SOCKS5 CONNECT requests only
- open remote connections through SSH based on the requested destination
- reject unsupported SOCKS commands with clear errors

#### Runtime lifecycle

- one foreground supervisor owns the SSH session and the configured listeners
- local listeners accept connections concurrently
- each accepted connection gets its own SSH direct stream channel
- `Ctrl+C` shuts down listeners and active channels cleanly
- if the SSH session dies, the command exits and reports the failure

## Data Model

Add a forward definition model associated with profiles.

Suggested forward record fields:

- `profile_name`
- `name`
- `kind`: `local` or `socks`
- `bind_host`
- `bind_port`
- `target_host` optional for `socks`, required for `local`
- `target_port` optional for `socks`, required for `local`
- `description` optional

Storage approach:

- persist forward definitions in SQLite alongside existing profile metadata
- keep forwards non-secret
- preserve current secret storage boundaries for passwords, imported keys, and key passphrases

## Architecture

Keep the shared SSH bootstrap and add three focused areas:

- `doctor`
  - check runners
  - result aggregation
  - output formatting
- `forward`
  - config parsing
  - persistence
  - listener/runtime supervisor
- `copy`
  - resume-aware planning
  - progress policy
  - transfer summary generation

The current SSH/auth/host-key flow remains shared:

- TOFU host-key verification
- auth mode precedence
- keepalive configuration
- session setup and teardown semantics

## Error Handling

### Doctor

- failed checks should be reported individually instead of aborting at the first local issue
- live checks may short-circuit where later checks depend on earlier ones
- errors should be mapped to actionable messages such as agent unavailable, host key missing/mismatched, remote unreachable, or no usable auth material

### Copy

- reject `--resume` with recursive copy
- reject `--resume` when destination state is incompatible
- include source, destination, and transfer direction in failure messages where practical

### Forwarding

- fail fast on invalid bind specs or duplicate forward names
- fail startup if any requested listener port cannot bind
- report per-connection proxy failures without crashing the whole forward process unless the SSH session itself is lost

## Testing Plan

### CLI tests

- `doctor` help and parsing
- `forward` add/list/remove/run help and parsing
- `copy --resume` and `copy --progress` parsing

### Unit tests

- doctor result classification and exit mapping
- forward spec parsing and validation
- copy resume planning rules
- stored forward persistence

### Integration tests

Linux OpenSSH integration tests should be extended to cover:

- `doctor <profile>` successful live checks
- `doctor <profile>` host-key mismatch and auth-mode failures
- resumable upload
- resumable download
- forwarding a local TCP listener through SSH to a remote endpoint
- SOCKS5 connect through SSH to a remote endpoint

### Platform validation

- add Windows coverage for agent detection and bind semantics where practical
- add macOS validation for doctor and foreground forwarding flows

## Rollout Notes

- keep the current top-level command layout intact
- add `forward` as a new subcommand family rather than overloading `open`
- keep `copy` behavior backward-compatible unless `--resume` or `--progress` is explicitly requested
- avoid shipping background tunnel management until foreground forwarding proves stable
